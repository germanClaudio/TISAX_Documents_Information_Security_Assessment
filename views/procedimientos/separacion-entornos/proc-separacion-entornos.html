Procedimiento de Separación de Entornos de Desarrollo

Código: MC522-IT-1
Última revisión: 25/8/2025
Rev. N°: 0
Cumple: TISAX AL2, ISO 27001:2022

1 OBJETIVO

Garantizar que los entornos de desarrollo, pruebas y producción estén adecuadamente separados para:

  • Proteger los datos operativos críticos (ej.: financieros en SIP 4.0).
  • Prevenir modificaciones no controladas en sistemas productivos.

Este procedimiento se alinea con los requisitos de ISO/IEC 27001:2022, TISAX y clientes automotrices.

2 EVALUACIÓN DE RIESGOS PARA SEPARACIÓN DE ENTORNOS

A. Alcance:

Aplicable a:

  • SIP 4.0 (entorno productivo).
  • Aplicación interna desarrollada (entorno desarrollo/pruebas).
  • Proveedores externos (acceso a entornos).

B. Proceso:

  1. Identificación de Activos Críticos:
    • Clasificar datos/sistemas según sensibilidad (ej.: SIP 4.0 = "Crítico").
  2. Análisis de Riesgos:
    • ¿Qué pasaría si el programador modifica directamente producción? → Riesgo alto.
  3. Determinar Necesidad de Separación:
    • SIP 4.0: Entorno productivo aislado (sin acceso directo del programador).
    • App interna: Entorno desarrollo/pruebas con datos sintéticos.

C. Documentos de Referencia:

  • M522-IT-2 Evaluación de Riesgos (Matriz de riesgos).

3 SEGMENTACIÓN DE ENTORNOS

A. Alcance:

  • Implementación de controles técnicos y organizativos.

B. Proceso:

  1. Separación Física/Lógica:
    • SIP 4.0 (Producción):
      • Servidor AWS con puertos diferenciados en url para entorno de pruebas (:9080) y desarrollo (:8080).
      • Credenciales administrador solo para Gerencia.
    • App Interna (Desarrollo/Pruebas):
      • Instancia separada en servidor local o cloud (ej.: GitHub).
      • Usar datos falsos (no copiar BD real de SIP).
  2. Control de Accesos:
    • Programador part-time:
      • Acceso solo a entorno desarrollo.
      • Cero permisos en SIP 4.0.
    • Proveedor ERP:
      • Acceso a producción supervisado (sesiones grabadas).

C. Documentos de Referencia:

  • MC522-IT-1 Política de Segmentación de Entornos (Política de accesos).

4 GESTIÓN DE CAMBIOS ENTRE ENTORNOS

A. Alcance:

  • Movimiento de código/configuraciones desde desarrollo → producción.

B. Proceso:

  1. Desarrollo:
    • Programador trabaja solo en entorno local con Git (ej.: GitHub Private).
  2. Pruebas:
    • Validación en entorno aislado con datos de prueba.
    • Aprobación por responsable de área (ej.: jefe de producción).
  3. Despliegue a Producción:
    • Realizado por personal designado (no el programador).
    • Backup obligatorio pre-despliegue.

Recomendaciones para Implementación

  1. Coste Cero:
    • Usar máquinas virtuales locales (VirtualBox) para desarrollo/pruebas.
  2. Proveedores ERP:
    • Exigir contrato con cláusulas de acceso supervisado a producción.
