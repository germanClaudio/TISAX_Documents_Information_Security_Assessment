Procedimiento de Aplicación de Parches

Código: P525-IT-2
Última revisión: 25/8/2025
Rev. N°: 0
Cumple: TISAX AL2, ISO 27001:2022

1 OBJETIVO

Establecer un proceso sistemático y controlado para la gestión e implementación de parches de seguridad en todos los sistemas de información de PRODISMO SRL, con el fin de mantener la integridad, confidencialidad y disponibilidad de los activos de información, cumpliendo con los controles de la norma ISO/IEC 27001:2022, así como con los requisitos de protección de activos críticos del marco TISAX.

2 ALCANCE

Este procedimiento aplica a:

  • Sistemas operativos: Windows Server, Linux
  • Aplicaciones empresariales: ERP SIP 4.0, suites ofimáticas, navegadores
  • Sistemas de seguridad: Antivirus, firewalls, sistemas de detección de intrusos
  • Infraestructura de red: Switches, routers, sistemas de almacenamiento
  • Servicios en la nube: Microsoft 365, Azure, servicios SaaS corporativos
  • Estaciones de trabajo: PCs de escritorio, portátiles corporativos

3 DEFINICIONES

  • Parche de Seguridad: Actualización software para corregir vulnerabilidades
  • Parche Crítico: Corrección para vulnerabilidades con CVSS ≥ 7.0
  • Ventana de Mantenimiento: Período autorizado para realizar cambios
  • Rollback: Procedimiento para revertir una actualización fallida
  • CAB (Change Advisory Board): Comité de evaluación de cambios

4 CLASIFICACIÓN DE PARCHES

4.1. Niveles de Criticidad

Nivel CVSS Descripción Plazo Máximo Aplicación
Crítico 9.0-10.0 Vulnerabilidad explotada activamente o de impacto severo 72 horas
Alto 7.0-8.9 Vulnerabilidad con alto impacto potencial 14 días
Moderado 4.0-6.9 Vulnerabilidad con impacto limitado 30 días
Bajo 0.1-3.9 Vulnerabilidad con impacto mínimo 90 días

5 PROCESO DE APLICACIÓN DE PARCHES

5.1. Fase 1: Identificación y Evaluación

Actividades:

  1. Monitoreo diario de fuentes oficiales (Microsoft, Linux vendors, CVE)
  2. Recepción de alertas del sistema de gestión de vulnerabilidades
  3. Evaluación de impacto en sistemas específicos
  4. Clasificación según matriz de criticidad

Criterios de Evaluación:

  • Criticidad de la vulnerabilidad (CVSS)
  • Sistemas afectados y su exposición
  • Existencia de exploits públicos
  • Impacto operacional potencial

5.2. Fase 2: Planificación

5.2.1. Parches Críticos (Aplicación Emergencia)

  • Notificación inmediata al RSI y CAB
  • Evaluación expedita de impacto (máximo 4 horas)
  • Planificación de ventana de emergencia
  • Comunicación a usuarios afectados

5.2.2. Parches No Críticos (Aplicación Programada)

  • Inclusión en ciclo regular de parches
  • Programación en ventana de mantenimiento estándar
  • Comunicación con 5 días de antelación mínima

5.3. Fase 3: Pruebas

Entorno de Testing:

  • Réplica representativa de sistemas productivos
  • Pruebas de funcionalidad básica y específica
  • Verificación de compatibilidad con aplicaciones críticas
  • Duración mínima de pruebas: 24 horas para parches críticos

Criterios de Aprobación para Producción:

  • Sin errores críticos en pruebas
  • Verificación de funcionalidad de negocio
  • Confirmación de resolución de vulnerabilidad

5.4. Fase 4: Implementación

Pre-Implementación:

  • Backup completo de sistemas (incluido estado del sistema)
  • Verificación de plan de rollback
  • Comunicación final a usuarios

Implementación:

  • Aplicación según ventana autorizada
  • Monitorización continua durante el proceso
  • Verificación punto por punto de sistemas críticos

5.5. Fase 5: Verificación Post-Implementación

Actividades Inmediatas (0-2 horas):

  • Verificación de servicios críticos
  • Confirmación de aplicación correcta del parche
  • Monitoreo de indicadores de performance

Actividades Posteriores (24-48 horas):

  • Escaneo de vulnerabilidades para confirmar remediación
  • Revisión de logs de sistema para detectar anomalías
  • Encuesta rápida a usuarios clave

5.6. Fase 6: Documentación y Cierre

Registros Obligatorios:

  • F525-IT-3 Formulario de Aplicación de Parches
  • Evidencias de testing y verificación
  • Incidencias reportadas y resoluciones
  • Lecciones aprendidas

6 VENTANAS DE MANTENIMIENTO

6.1. Horarios Establecidos

Tipo de Sistema Ventana Regular Ventana Emergencia
Sistemas Críticos (ERP SIP 4.0) Sábado 22:00-06:00 Cualquier hora con aprobación CAB
Servidores No Críticos Viernes 17:00-20:00 Fin de semana
Estaciones de Trabajo Actualizaciones automáticas fuera horario laboral Notificación push con reinicio programado

7 PROCEDIMIENTO DE ROLLBACK

7.1. Criterios para Activación

  • Caída de servicios críticos por más de 15 minutos
  • Errores graves no resueltos en 2 horas
  • Solicitud del dueño del sistema por impacto operacional severo

7.2. Pasos del Rollback

  1. Notificación inmediata al RSI y CAB
  2. Ejecución de backup pre-parchado
  3. Restauración de servicios críticos
  4. Análisis de causas del fallo
  5. Documentación completa del incidente

8 HERRAMIENTAS Y AUTOMATIZACIÓN

8.1. Herramientas Autorizadas

  • Windows: WSUS, Microsoft Endpoint Configuration Manager
  • Linux: Ansible, Satellite, yum/apt repositories internos
  • Third-party: ManageEngine Patch Manager, Ivanti Security Controls
  • Cloud: Azure Update Management, AWS Systems Manager

8.2. Configuraciones Estándar

  • Descarga automática de parches críticos
  • Aprobación manual para implementación
  • Reporting automático de estado de parches
  • Integración con sistema de ticketing (STI)

9 MÉTRICAS Y REPORTE

9.1. Indicadores Clave de Performance

  • Tiempo promedio de aplicación: Desde publicación a implementación
  • Tasa de cumplimiento: Porcentaje de parches aplicados en plazo
  • Tasa de éxito: Parches aplicados sin incidencias
  • Cobertura: Porcentaje de sistemas bajo gestión de parches

9.2. Reporte a Dirección

  • Reporte Mensual: Estado de parches críticos y métricas clave
  • Reporte Trimestral: Tendencias, efectividad del proceso
  • Reporte Anual: Evaluación completa para revisión de dirección

10 GESTIÓN DE EXCEPCIONES

10.1. Solicitud de Excepción

  • Justificación técnica/business para no aplicar parche
  • Evaluación de riesgo formal y plan de mitigación
  • Aprobación requerida del RSI y CAB
  • Revisión máxima cada 90 días

11 REVISIÓN Y AUDITORÍA

11.1. Revisiones Periódicas

  • Revisión Mensual: Efectividad del proceso operativo
  • Revisión Trimestral: Actualización basada en lecciones aprendidas
  • Revisión Anual: Evaluación completa del procedimiento

11.2. Auditorías de Cumplimiento

  • Auditorías internas semestrales
  • Verificación de evidencias de aplicación
  • Inclusión en evaluaciones TISAX y auditorías ISO/IEC 27001:2022
