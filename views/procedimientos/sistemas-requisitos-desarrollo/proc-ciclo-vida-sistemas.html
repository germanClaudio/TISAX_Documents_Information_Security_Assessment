Procedimiento de Seguridad en el Ciclo de Vida de Sistemas IT

Código: P531-IT-1
Última revisión: 25/8/2025
Rev. N°: 0
Cumple: TISAX AL2, ISO 27001:2022

1 OBJETIVO

Integrar controles de seguridad en todas las fases (diseño, adquisición, desarrollo, pruebas) de sistemas IT nuevos o existentes, garantizando la protección de datos críticos (ej.: datos financieros en SIP4.0, propiedad intelectual). Este procedimiento se alinea con los requisitos de ISO/IEC 27001:2022 y TISAX.

2 DISEÑO SEGURO

A. Alcance:

  • Aplicaciones internas, integraciones con ERP SIP 4.0, infraestructura cloud/on-premise.

B. Proceso:

1. Requisitos de Seguridad:

  • Definir durante el análisis de necesidades (ej.: cifrado AES-256 para datos financieros).
  • Alinear con normativas (TISAX, GDPR).

2. Arquitectura:

  • Aplicar principio de mínimo privilegio en accesos.
  • Segmentar componentes críticos (ej.: módulo de pagos en VLAN separada).

C. Documentos de Referencia:

  • MC531-IT-1 Política de Requisitos de Seguridad (Checklist de diseño seguro).

3 ADQUISICIÓN DE SISTEMAS

A. Alcance:

  • Software comercial, hardware, servicios cloud.

B. Proceso:

1. Evaluación de Proveedores:

  • Verificar certificaciones de seguridad (ISO27001, SOC 2).
  • Exigir cláusulas contractuales de cumplimiento TISAX.

2. Pruebas Preliminares:

  • Escanear vulnerabilidades con OWASP ZAP (para software).
  • Validar soporte de MFA y cifrado.

C. Documentos de Referencia:

  • P531-IT-2 Procedimiento de Evaluación de Proveedores (Criterios técnicos).

4 DESARROLLO SEGURO

A. Alcance:

  • Aplicaciones desarrolladas internamente (ej.: app de producción).

B. Proceso:

1. Prácticas de Codificación:

  • Validar inputs (evitar SQL injection).
  • No almacenar credenciales en código.

2. Revisiones:

  • Análisis estático de código (ej.: SonarQube).
  • Pentesting básico para módulos críticos.

C. Documentos de Referencia:

  • I531-IT-1 Guía de Desarrollo Seguro (Estándares de programación).

5 PRUEBAS DE SEGURIDAD

A. Alcance:

  • Sistemas antes de pasar a producción.

B. Proceso:

1. Tipos de Pruebas:

Automáticas:

  • Escaneo de vulnerabilidades (Nessus).

Manuales:

  • Verificar controles de acceso en SIP 4.0.
  • Simular ataques (ej.: fuerza bruta a API).

2. Aprobación:

  • Firmar checklist de cumplimiento (F531-IT-1 Checklist de Pruebas).

6 GESTIÓN DE CAMBIOS

A. Alcance:

  • Actualizaciones de sistemas existentes (ej.: nuevo módulo en SIP 4.0).

B. Proceso:

1. Impacto en Seguridad:

  • Evaluar riesgos con matriz CVSS (M141-IT-2 Matriz de Riesgos) Calculadora CVSS v4.0.

2. Validación Post-Cambio:

  • Re-ejecutar pruebas críticas (ej.: roles de acceso).

Recomendaciones para Implementación

  1. Para SIP 4.0:
    • Exigir al proveedor informes de pentesting anuales.
  2. Costo Cero:
    • Usar OWASP ZAP y SonarQube (gratuitos) para pruebas.

Integración:

  • Vincular MC531-IT-1 Política de Requisitos de Seguridad, con el ciclo de vida de proyectos (ej.: fases en Jira).
